# Cloud Broadcast — Discord 限定配信 MVP 実装仕様書

## 0. この文書について

Discord のボイスチャンネルの会話を、BGM とミックスして YouTube Live へアーカイブ配信する仕組みの MVP 実装仕様。ブレストで確定した設計判断を実装可能な粒度まで落としたもの。

対象読者は実装担当者（本人）。将来の汎用化（入力の 3 層分類、WebRTC ingest、ローカルエージェント、映像合成の作り込み）は本 MVP の範囲外とし、必要な箇所でのみ拡張余地として触れる。

---

## 1. 背景と目的

### 1.1 解決する課題

現行の Discord 配信は、配信担当者のローカル環境に強く依存している。

- OBS で再生した BGM を仮想オーディオデバイス経由で Discord へ流し込む
- Discord ルームの会話を OBS のアプリケーション音声キャプチャで拾い、YouTube へアーカイブ配信する

この双方向の音声配線（OBS が「送り手」と「受け手」を兼ねる）＋仮想オーディオのルーティングが、他人には再現不能な属人化の正体。「あの人にしか配信できない」状態を生んでいる。

### 1.2 目的

音声の送出・受信・ミックスをすべて **Discord Bot** に畳み込み、ローカルの OBS・仮想オーディオデバイス・アプリケーション音声キャプチャを排除する。構成をコード化し、誰が起動しても同一の配信環境が立ち上がる状態にする。

### 1.3 非目標（MVP では扱わない）

- 映像合成・シーン切替・話者切替（現行運用でも行っていない）
- 複数話者の映像表示、発話リング等の映像演出
- ブラウザ / ローカルエージェントからの入力（入力の汎用化）
- 配信の同時多重運用（当面 1 配信）

## 2. スコープ

### 2.1 機能スコープ

1. Bot が指定ボイスチャンネルに参加する
2. 事前に選択した BGM プレイリストをループ再生する（本番中の手動操作なし）
3. ボイスチャンネルの会話（全参加者）を受信する
4. 会話 + BGM を 1 トラックにミックスする
5. 静止画映像 + ミックス音声を YouTube Live へ RTMPS 送出する
6. 配信の開始 / 停止を制御する

### 2.2 音声フロー

```
                 ┌─────────────────────────────┐
                 │        メディアプレーン        │
 Discord VC ─┐   │  ┌──────────────────────────┐ │
  (会話)      ├──►│  │ 受信: 参加者ごとのOpus     │ │
             │   │  │  → デコード → 加算ミックス  │ │
 BGMファイル ─┼──►│  │ 手元のBGM(ループ) ────────┤ │──► YouTube Live
  (R2)       │   │  │  → ミックスに合流          │ │    (RTMPS)
             │   │  └──────────────────────────┘ │
 Bot再生 ────┘   │   ※Bot送出音(BGM)は受信に現れない│
 (BGMをVCへ)     └─────────────────────────────┘
```

補足: Discord の受信は参加者ごとに別ストリームで届き、**Bot 自身が送出している BGM は受信側に現れない**。そのため「受信した会話ストリーム + 手元の BGM ファイル」を加算すればループバックなしでクリーンにミックスできる。現行の「スピーカー出力の最終ミックスを拾う」方式より音質面でも有利。

## 3. アーキテクチャ

### 3.1 プレーン分離

| プレーン | 責務 | 実装 |
|---|---|---|
| 制御プレーン | 起動/停止トリガ、構成保管、Secret/OAuth 管理、状態管理 | Cloudflare（Workers / D1 / R2 / Secrets Store） |
| メディアプレーン | Discord 音声の受信・BGM 再生・ミックス・RTMPS 送出 | Cloudflare Containers（PoC-0 実測で確定・2026-07-03 / 13 章） |

分離の根拠: メディアプレーンは長時間の常駐接続と UDP 処理（後述）を要するため、Workers（リクエスト駆動・短命実行）では成立しない。制御プレーンは短命処理でよいため Workers に寄せ、メディアプレーンは Cloudflare Containers（`sleepAfter` で常駐可、アウトバウンド UDP は PoC-0 実測合格）に置く。

### 3.2 制御フロー

```
[トリガ] ──► Workers ──► コンテナ基盤API(起動) ──► コンテナ
                │                                    │
                │  構成YAML + 短命トークン を注入 ────┘
                │
                └──► D1(状態記録) / R2(BGM,構成) / Secrets(トークン元)
```

1. トリガ（4 章）が Workers のエンドポイントを叩く
2. Workers が D1 / R2 から構成を読み、YouTube アクセストークンを用意する
3. Workers がコンテナ基盤の API を呼び、コンテナを起動する
4. 起動時に構成と短命トークン（ストリームキー等）を環境変数 / 引数で注入する
5. コンテナが Discord VC に参加し、YouTube への送出を開始する
6. 停止トリガでコンテナを停止 / 破棄する

### 3.3 プロトコル制約（設計の要）

- **Discord 音声 = UDP のみ。TCP フォールバックなし**。UDP が通らない環境では音声が物理的に成立しない（オール・オア・ナッシング）
- Discord Gateway（制御）= WebSocket / TCP
- YouTube 送出 = RTMPS / TCP（アウトバウンド単方向）
- したがってメディアプレーンの基盤は **アウトバウンド UDP を確実に捌けること** が絶対条件

> **基盤は Cloudflare Containers で決着（PoC-0 実測合格・2026-07-03）**。灰色領域だったアウトバウンド UDP は、deploy 済み CF ネットワーク上で STUN 往復（3 試行）と Discord voice 実接続（`VoiceConnectionStatus.Ready` 到達）を実測し、成立を確認した。inbound TCP/UDP 不可（公式明記）は変わらないが、Discord 音声は Bot 発のアウトバウンドなので影響しない。メディアプレーンも Cloudflare に寄せ、制御と同一スタックで構成する。詳細は `cloud-broadcast-poc0-result.md` / `cloud-broadcast-feasibility-check.md` A-1 を参照。9 章の常駐基盤候補は移行時の保険に位置づけを変更。

## 4. 配信ライフサイクル

### 4.1 起動トリガ（要決定 → 一次案: Discord スラッシュコマンド）

| 方式 | 概要 | 備考 |
|---|---|---|
| Discord スラッシュコマンド | `/broadcast start` を Workers が webhook で受ける | Discord 内で完結。スポット運用に素直。**一次案** |
| Cron Triggers | Workers の定期実行で自動起動 | 定時配信向け。MVP では不要 |
| 手動 UI | 管理画面のボタン | 実装コスト高。MVP では過剰 |

MVP は Discord スラッシュコマンドを一次実装とする。Cron・UI は将来拡張。

### 4.2 状態遷移

```
idle ──start──► provisioning ──ready──► live ──stop──► stopping ──► idle
                    │                     │
                    └──error──► failed ◄──┘
```

- `provisioning`: コンテナ起動〜VC 参加〜YouTube 接続確立まで
- `live`: 配信中
- `stopping`: VC 離脱・送出停止・コンテナ破棄
- `failed`: 起動失敗 or 配信中の致命的障害

状態は D1 に記録する。1 配信 1 レコード。

### 4.3 障害時の挙動（MVP 割り切り）

自宅作業員で物理アクセス不可・現地対応不可という制約のもと、MVP では以下とする。

- 配信中に Bot が落ちた場合: **自動復旧は実装しない**。状態を `failed` に記録し、通知のみ行う（Discord へメッセージ等）
- Gateway の一時切断: `@discordjs/voice` / ライブラリの標準再接続に委ねる範囲まで
- 恒久的なウォッチドッグ・自動再起動は将来拡張

### 4.3.1 配信中断の即時検知・自動リカバリ（要件）

物理アクセス不可の制約下では「中断に気づけない」ことが最大のリスクになるため、以下を**MVP 必須要件**とする（4.3 の「自動復旧しない」より上位。中断の放置は許容しない）。

- **R1（即時検知・必須）**: `live` 状態の配信が `stop` コマンド以外の理由で中断した場合、制御プレーンが**即座に検知**し、状態を `failed` に遷移させてオペレータへ通知する（検知遅延はワークロード上分単位を上限とする）。検知手段は、メディアコンテナ→制御プレーンのハートビート断、またはコンテナ処理リクエストの異常終了、または YouTube ingest 状態のポーリングのいずれかで実装する。
- **R2（自動リトライ・推奨→可能なら必須）**: 中断検知時、同一配信構成で provisioning を自動再実行し（VC 再 join → FFmpeg 再起動 → **同一ストリームキーで RTMPS 再接続**＝同じ YouTube 配信枠に復帰）、配信を自動再開する。再試行回数・バックオフ上限を設け、超過時のみ `failed` 確定 + 通知。
- **設計含意（重要）**: 上記を満たすため、**ブロードキャストの生存を単一の in-flight HTTP リクエストに束ねてはならない**。制御プレーンからの start/stop コマンドで駆動される常駐タスクとしてコンテナ側で走らせ、生存は制御プレーンが状態監視する構成にする（PoC-1〜3 は検証簡便化のため HTTP リクエスト内で実行しているが、本実装はこの分離を行う）。

**根拠となる中断要因（実測・ドキュメント確認済み、2026-07-06）**:
- Cloudflare Workers は HTTP リクエストの実行時間に上限を設けない（`workers/platform/limits`「HTTP request: No limit / DO: Unlimited（接続維持・pending I/O の間）」）。よって 30 分配信自体はサポートされる。
- ただし**ランタイム更新時、in-flight リクエストには 30 秒の猶予のみ与えられ、超過すると terminate される**（更新は週に数回。長時間リクェストが更新と重なった場合に発生。CF は "very unlikely" と記載）。単一リクエスト束ねだとこの稀な更新衝突で配信が切れる。R1/R2 と上記「常駐タスク化」でこれを検知・自動復帰の対象にする。
- 併せて、戻り UDP（音声受信）の長時間 NAT マッピング維持・`@discordjs/voice` の再接続も R1 の監視対象に含める。

## 5. 構成スキーマ（構成 as Code）

### 5.1 方針

配信構成を宣言的に記述しリポジトリ / R2 で管理する。**Secret（ストリームキー・トークン）はスキーマに含めず、実行時に注入**する。これが属人化排除の本丸。

### 5.2 スキーマ（MVP）

```yaml
version: 1
broadcast:
  title: "Discord Room Archive"

  discord:
    guild: "<guild_id>"
    channel: "<voice_channel_id>"
    tokenRef: secret://discord/bot-token   # 実体は注入
    capture: mixed                          # 全参加者を1トラックにミックス
    bgm:
      # 本番前に選択。本番中はループ固定
      playlist:
        - "r2://bgm/track1.mp3"
        - "r2://bgm/track2.mp3"
      loop: true

  output:
    platform: youtube
    keyRef: secret://youtube/stream-key     # 実体は注入
    video:
      source: static-image
      image: "r2://assets/standby.png"      # 音声主体のため静止画
      resolution: "1920x1080"
      fps: 15                                # 静止画なので低fpsで可
      bitrate: "1000k"
    audio:
      codec: aac
      bitrate: "160k"
      sampleRate: 48000
```

### 5.3 含める / 含めない

- **含める**: タイトル、Discord 参加先、capture モード、BGM プレイリスト、送出先種別、映像/音声のエンコード設定
- **含めない（Secret / 実行時注入）**: Bot トークン、YouTube ストリームキー、OAuth トークン、セッション固有値

### 5.4 検証

TypeScript + Zod でスキーマをバリデートする。`tokenRef` / `keyRef` は `secret://` 形式の参照のみを許可し、実体値がスキーマに混入していたら弾く（属人化・秘匿情報混入の防止）。

## 6. メディアプレーン実装

### 6.1 技術スタック

- ランタイム: Node.js + TypeScript
- Discord 音声: `@discordjs/voice`（再生・受信・Opus 処理）
- ミックス / エンコード / 送出: FFmpeg
- コンテナ: Docker イメージ（Bot + FFmpeg + 依存を全て焼き込む）

> `@discordjs/voice` の受信 API は過去に仕様変更歴がある。実装着手時に現行の Discord Developer Docs（Voice の章）と `@discordjs/voice` の最新版で受信・再生の API を確認すること。

### 6.2 処理構成

1. **参加**: `tokenRef` で解決した Bot トークンで Gateway 接続 → 指定 VC へ join
2. **BGM 再生**: プレイリストをループで VC へ送出（`AudioPlayer`）
3. **会話受信**: VC の各参加者の Opus ストリームを受信（`VoiceReceiver`）→ デコード（PCM）
4. **ミックス**: 受信 PCM 群 + BGM の PCM を加算ミックスして 1 トラックの PCM を生成
5. **エンコード**: FFmpeg で「静止画（ループ映像）+ ミックス PCM」を H.264 + AAC に多重化
6. **送出**: FFmpeg から `rtmps://` で YouTube Live ingest へ push

### 6.3 ミックスの要点

- Bot 自身の BGM 送出は受信ストリームに含まれないため、BGM は受信を待たず手元 PCM から直接合流させる（二重取り込み・ループバックなし）
- `capture: mixed` は「受信した全参加者ストリームの加算」を意味する。将来 `perUser`（話者別保持）に拡張する余地を型として残してよいが、MVP は `mixed` のみ実装
- サンプルレート / チャンネル数を受信・BGM・出力で揃える（48kHz を基準）

### 6.4 コンテナへの注入（起動時）

環境変数または起動引数で以下を渡す。イメージには焼かない。

- 構成 YAML（または D1/R2 上の構成 ID）
- Discord Bot トークン（短命に扱う）
- YouTube ストリームキー / アクセストークン

BGM は R2 に置き、起動時に署名付き URL 経由でコンテナが取得するか、Workers が取得して渡す。

## 7. YouTube 送出

### 7.1 プロトコル

YouTube Live ingest は **RTMPS（TCP）** を標準とする。FFmpeg から `rtmps://` で送出する。

### 7.2 配信枠の用意（要決定）

| 方式 | 概要 | MVP 判断 |
|---|---|---|
| 手動 | YouTube Studio で配信枠を作りストリームキーを構成へ手貼り | 実装ゼロ。**MVP はこれで可** |
| API 自動作成 | YouTube Data API v3（`liveBroadcasts` / `liveStreams`）で枠とキーを自動生成 | 将来拡張。OAuth 運用が付随 |

MVP は手動でストリームキーを用意し `secret://youtube/stream-key` に格納する方式を一次とする。自動化は次段階。

### 7.3 OAuth（自動化する場合）

自動作成に進む場合、refresh token を Cloudflare Secrets Store / D1 に保管し、Workers がアクセストークンを更新する。コンテナには短命のアクセストークン / ストリームキーのみ渡す。過去の YouTube Data API v3 + OAuth 2.0 の知見を流用可能。

## 8. Secret 管理

| Secret | 保管先 | コンテナへの渡し方 |
|---|---|---|
| Discord Bot トークン | Cloudflare Secrets Store | 起動時に環境変数で注入 |
| YouTube ストリームキー | Cloudflare Secrets Store | 起動時に環境変数で注入 |
| YouTube refresh token（自動化時） | Cloudflare Secrets Store / D1 | コンテナには渡さず、Workers 側で access token に交換して注入 |

原則: 長命の Secret は制御プレーンに留め、メディアプレーンには短命・最小限のものだけ渡す。

## 9. ホスティングとコスト

### 9.1 前提ワークロード

- 月数回 × 最長 30 分 = 月間稼働 約 2〜3 時間
- GPU 不要（静止画）、CPU 軽量（音声デコード + ミックス + AAC）、帯域極小（音声 160k + 静止画）

### 9.2 コスト帯（月額上限 n、2026-07 時点調査。採用時に再確認）

基盤は Cloudflare Containers で確定（PoC-0）したため、コストは Cloudflare 側に統合される。

**月額上限 n（PoC-3 で確定、2026-07-06 現行料金で算出）**:

- 確定インスタンス: `instance_type = standard-3`（2 vCPU / 8 GiB / 16 GB。PoC-3 で N_target=2 が定常 CPU 35% と大きな余裕で合格）
- 稼働: §9.1 のワークロード（月数回 × 30 分 ≈ 月 2〜3 時間）
- 現行 Containers 料金（Workers Paid、docs 確認）: CPU $0.00002/vCPU-秒（**active 使用分のみ**課金）、メモリ $0.0000025/GiB-秒、ディスク $0.00000007/GB-秒、egress $0.025/GB（NA&EU）。月間 included: CPU 375 vCPU-分、メモリ 25 GiB-時、ディスク 200 GB-時、egress 1TB。
- 算出: 月 3 時間・**CPU 100% 想定（保守）**でも CPU 21,600 vCPU-秒 < 22,500 included、メモリ 86,400 GiB-秒 < 90,000 included、ディスク・egress とも included 内 → **コンテナ従量 = $0**。月 5 時間×100% でも従量 +$0.41、月 10 時間×100% でも +$1.49 に留まる。
- **∴ 月額上限 n ≈ Workers Paid 基本料 $5（≈ ¥800）**。想定ワークロードではコンテナ従量は無料枠内で実質 0、制御プレーンの Worker/DO リクエストも僅少。「上限を言い切れる」（§9.3）を満たす。

以下の常駐基盤の比較は、Cloudflare から移行が必要になった場合の保険として残す。

| 帯 | サービス | n（月額） | 備考 |
|---|---|---|---|
| 最安 | Oracle Cloud Always Free | 0 円 | 容量枯渇・アカウント凍結リスク。本番は不安定 |
| 中間 | Google Cloud Run | 0〜数十円 | scale-to-zero と常駐 UDP の相性が悪く要検証 |
| 従量 | Fly.io Machines | 約 40 円（最悪 340 円） | 停止中課金なし。常駐 UDP 向きで素直 |
| 定額 | Hetzner CX22 | 約 850 円（固定上限） | 完全予測可・安定。保守工数は自前 |

### 9.3 会社利用の判断軸

- コストの絶対額より「上限 n を言い切れる（予測可能性）」が要件
- 物理アクセス不可のためオンプレ / 自宅サーバーは不可
- **基盤は Cloudflare Containers で決着**（PoC-0 合格・2026-07-03）。上表の常駐基盤候補は移行時の保険に位置づけを変更
- 各候補の技術的評価（UDP との相性等）は tech ナレッジ `cloud-broadcast-hosting-cost-tiers` を参照

## 10. 規約・コンプライアンス

- ボイスチャンネルの会話を録音・配信するため、参加者への録音・配信の明示が前提（取得済み）
- Bot 方式では Bot の存在自体が録音の可視化になる。Bot 命名・チャンネル通知等で明示を運用担保する
- Discord のデベロッパーポリシー上、音声受信の扱いは実装着手時に現行規約で確認すること

## 11. 却下した設計判断

ブレスト過程で検討し、不採用としたもの。同じ議論の再発防止のために記録する。

| 却下したもの | 却下理由 |
|---|---|
| 複数人**同時**編集（CRDT / 排他制御 / 権限モデル） | 当初「複数人運用」と捉えたが、真の課題は「誰でも同じ環境で配信できる（属人化排除）」であり同時並行制御ではなかった。並行制御は不要 |
| 配信ノウハウの環境への埋め込み（最適プリセット自動選択 / 開始前ガードレール / 異常検知でオペレーションを代替） | 課題を「熟練者の暗黙知の属人化」と取り違えた案。真の課題は「もらった設定が環境差で動かない」再現性の問題で、ノウハウの形式知化は的外れ |
| ブラウザ WebRTC でのアプリケーションキャプチャ / 特定アプリ音声の取得 | `getDisplayMedia` の粒度は画面/ウィンドウ/タブまで。OS のウィンドウ合成をバイパスしてプロセス単位で映像/音声を抜くのは Web サンドボックス内で原理的に不可能 |
| ローカルの仮想オーディオデバイス経由での音声取り回し | まさに現行の属人化の温床。Bot に音声を集約する本 MVP の目的と正面衝突するため排除 |
| アプリごとの専用プラグイン化 | 分類軸は「どのアプリか」ではなく「どういう取得経路か」。アプリ単位で専用実装すると無限に増え破綻する（取得経路による層分けが正） |
| 自宅 / オンプレでの実行 | 最も安価（電気代のみ）だが、自宅作業員で物理アクセス不可・障害時に現地対応できない。かつ自宅依存＝属人化への逆戻り |
| 「常時無料（0 円絶対）」を前提とした基盤選定 | 会社利用では絶対額より「上限 n を言い切れる（予測可能性）」が要件。0 か 1 かではなく見積もれることが本質。無料枠のみに固執すると常駐 UDP との相性で破綻するか、凍結・容量枯渇リスクを抱える |
| メディアプレーンを Cloudflare Workers で実装 | Workers はリクエスト駆動・短命実行前提。数十分の常駐ボイス接続・生 UDP・FFmpeg のネイティブ重処理と実行モデルが噛み合わない（＝メディアプレーンを分離する動機そのもの） |

> Cloudflare Containers は PoC-0（2026-07-03）でアウトバウンド UDP 音声の成立を実測確認し、**メディアプレーン基盤として採用**した（13 章）。

## 13. 実装順序（推奨）

1. ✅ **PoC-0（完了・合格 2026-07-03）**: Cloudflare Containers でのアウトバウンド UDP 音声の成立を実測確認した（STUN 往復 3 試行 + `@discordjs/voice` 実接続で `VoiceConnectionStatus.Ready` 到達）。メディアプレーンも Cloudflare に寄せることが確定（`cloud-broadcast-poc0-result.md`）
2. **PoC-1（次工程）**: CF Containers 上でメディアパイプライン end-to-end（VC 参加 → 会話受信 → 静止画 + AAC で RTMPS 送出）を通す（`cloud-broadcast-poc-plan.md`）
3. **PoC-2**: BGM ループ再生と会話のミックス（不連続受信の連続化・クロック駆動ミキサ）を追加
4. **PoC-3**: 負荷計測でマシンサイズとコスト上限を確定
5. 構成スキーマ（Zod）と Secret 注入を実装
6. Workers による起動 / 停止制御（Discord スラッシュコマンド）と D1 状態管理
7. 障害時通知、運用面の仕上げ

ステップ 1 の UDP 実証で基盤は確定済み。以降のステップは基盤選定に依存しない（上位設計に影響しない）。
