# クラウド上に Discord 雑談の配信環境を構築する

## 課題

（だいたい）毎週月曜昼 12 時から Discord でやってる雑談配信は、:@cancer: のローカル環境に強く依存していた。現行の構成は次の通り。

- OBS で再生した BGM を、仮想オーディオデバイス経由で Discord へ流し込む
- Discord ルームの会話を、OBS のアプリケーション音声キャプチャで拾って YouTube へ流す

OBS が「送り手」と「受け手」を兼ね、その間を仮想オーディオデバイスでルーティングしている。この双方向の音声配線は他人の環境では再現できない。実際、:@cancer: が休みのときに :@geckotang: に配信を頼んだが、この環境を再現できなかった。属人化の実体はこれである。

課題は属人化の排除、つまり誰でも同じ環境で配信できるようにすることである。具体的には「あの人からもらった OBS プロファイルを使ったのに配信できなかった」を無くす。

これはノウハウ共有の問題ではない。OBS プロファイルは「設定」だけを持ち、「それが前提とする環境」を持たない。設定が外部参照している環境依存は次の通り。

- キャプチャ対象のデバイス名
- ウィンドウ・アプリ指定
- BGM やオーバーレイの絶対パス
- プラグイン依存
- エンコーダ（NVENC 前提など）

渡した先の環境が違えば、これらの参照が満たされず動かない。課題の実体は「設定を渡しても環境が違うと動かない」という再現性の問題である。

## 方針

解決策としては、設定ではなく動く環境に誰でもアクセスできるようにすること、とした。プロファイルとその依存環境が丸ごと揃った状態をクラウドに固定し、開けば誰でも同じに動く。

この方針の本質的な価値は、「なぜ動かないか」の原因究明を放棄できる点にある。個別トラブルシュートを人間がやる限り属人化は消えない。差異が生まれる余地そのものを消せば、原因が何であれ関係なくなる。具体的には、ローカルに残す依存を物理入力（カメラ・マイク）だけに絞り、それ以外（アプリ・プラグイン・アセット・エンコーダ・OS 差）をすべてクラウド側で固定する。

今回の Discord 雑談配信はその最初の具体形で、音声の送出・受信・ミックスをすべて Discord Bot に畳み込み、OBS と仮想オーディオデバイスを排除する。以降は、この Bot をどこで・どう走らせるかという実装の話になる。

## アーキテクチャ

配信システムを性質の違う 2 つのプレーンに分離した。

| プレーン | 責務 | 実装 |
|---|---|---|
| 制御プレーン | 起動/停止トリガ、構成保管、Secret 管理、状態管理 | Cloudflare Workers / D1 / R2 / Secrets Store |
| メディアプレーン | Discord 音声の受信・BGM 再生・ミックス・RTMPS 送出 | Cloudflare Containers |

分離の理由は実行モデルの違いにある。制御プレーンは「起動トリガを受けて状態を書く」だけの短命処理で、リクエスト駆動の Workers に素直に乗る。一方メディアプレーンは、数十分の常駐ボイス接続・生 UDP の処理・FFmpeg のネイティブ重処理を要求し、リクエスト駆動・短命実行前提の Workers とは実行モデルが相容れない。そのためメディアプレーンだけをコンテナに分離した。これは Cloudflare Containers が想定する「Worker がオーケストレータ、コンテナは呼ばれて働く実行体」という思想にも沿う。

音声フローは次の通り。

```
 Discord VC ──(参加者ごとの Opus)──┐
                                    ├─► デコード → 加算ミックス ─► FFmpeg ─► YouTube (RTMPS)
 BGM ファイル (R2) ─(手元 PCM)──────┘         ↑
 Bot 再生 (BGM を VC へ) ────────────  ※Bot が送出する BGM は受信側に現れない
```

設計上の勘所は、「Discord の受信は参加者ごとに別ストリームで届き、Bot 自身が送出している BGM はその Bot の受信側には現れない」という挙動である。これが成り立つなら、ミックスは「受信した会話ストリーム＋手元の BGM ファイル」を加算するだけで済み、ループバック（自分の出した音を自分で拾い直す二重取り込み）が原理的に発生しない。現行の「スピーカー出力の最終ミックスを拾う」方式より音質面でも有利になる。この前提は実機で確認した。

## 検証

この構成で最初に確認すべき点は 2 つある。Cloudflare Containers から Discord Voice に接続できるか、YouTube の RTMPS に接続してライブ配信を継続できるかである。前者は outbound UDP、後者は長時間の outbound TCP 接続の問題になる。

Discord Bot の Gateway は WebSocket/TCP だが、Voice は UDP のみで、TCP フォールバックがない。Cloudflare Containers は inbound TCP/UDP が直接使えないことは公式に明記されている一方、outbound UDP については明記されていなかった。

そのため、`wrangler deploy` 後の Cloudflare 実機上で outbound UDP を測った。Node の `dgram` で公開 STUN サーバへ Binding Request を投げ、ホスト名指定・IP 直指定・別ベンダー STUN の 3 通りで binding-success を確認した。続けて `@discordjs/voice` + `sodium-native` で VC join し、`VoiceConnectionStatus.Ready` に到達した。これで UDP の IP discovery と AEAD 暗号化ネゴシエーションまで通ることが確認できた。

ほかに実機で確認した点は次の通り。

- 他参加者の Opus を受信し、PCM にデコードできる
- 1 体の Bot が BGM を VC へ送出しながら他者音声を受信できる
- Bot 自身が送出した BGM は、その Bot の受信側には混入しない
- RTMPS で YouTube に接続し、ライブ配信として送出を継続できる
- 発話と BGM が YouTube のアーカイブに残る
- `standard-3` で 1080p 静止画エンコード＋想定同時発話数の処理に余裕がある

これで、制御プレーンとメディアプレーンを Cloudflare に寄せる構成は実装可能と判断した。

ただし Cloudflare Containers は常時稼働 VM ではなく、アイドル時に停止するライフサイクルを持つ。Cloudflare も、インスタンスが一定期間動き続けることは保証していない。また Workers runtime の更新時には in-flight request が 30 秒の猶予後に終了されうるため、長時間配信では瞬断を前提に中断検知と再接続を設計に含める必要がある。詳細は Cloudflare の [Containers FAQ](https://developers.cloudflare.com/containers/faq/#how-long-can-instances-run-for-what-happens-when-a-host-server-is-shut-down) と [Workers limits](https://developers.cloudflare.com/workers/platform/limits/#duration) を参照。

## Cloudflare Containers の運用知見

新イメージが反映されない問題にいちばん時間を取られた。これは根本原因を特定できていない。分かっているのは、稼働中の Durable Object シングルトンインスタンスが古いコンテナイメージのまま残り、`wrangler deploy` や `--containers-rollout immediate` を実行してもリクエストが旧コードへ届き続けた、という観測事実までである。

`sleepAfter` を長くしたことが原因とは言い切れない。ただし、旧インスタンスの居座りを長引かせた可能性はある。長時間テスト中に寝ないよう延ばしたくなるが、in-flight リクエストがある限りインスタンスは起き続けるので、短め（数分）で足りる。

原因は不明なままだが、回避策はあった。確実だったのは、`wrangler containers delete <appId>` でコンテナアプリごと削除してから `wrangler deploy` で新規作成する方法である。新アプリ ID・新インスタンスとして正しいイメージで起動する（初回 provisioning は数分かかる）。

## 運用上の前提

コンプライアンスについては、会話を録音・配信するため、参加者への録音・配信の明示が前提になる（同意は取得済み）。Bot 方式では Bot の存在自体が録音の可視化になる。

Discord は Bot による音声受信を公式にはドキュメント化・サポートしていない。今回の実測では成立したが、これは技術的な go/no-go というより継続性リスクとして保持する立場を取った。ライブラリ（`@discordjs/voice`）自体は discord.js org 管理の公式ライブラリで de-facto 安定しているが、Discord が音声の下回りを変えたとき一時的に追随待ちが発生しうる（2024-11 の暗号化強制移行が前例）。

## 残課題

PoC で確認したのは、Discord Voice への接続、YouTube への送出、BGM と会話のミックス、負荷の余裕までである。運用として必要な制御プレーンはまだ本実装していない。

残っている課題は以下。

- Workers での起動/停止
- D1 による状態管理
- Secret 注入
- 構成 as Code（Zod による構成スキーマ検証と、ストリームキー等を `secret://` 参照でしか書けなくする仕組み）
- 中断検知と通知（停止コマンド以外で live が切れたら検知する）
- 自動再接続（同一構成・同一ストリームキーで復帰する）
- 30 分耐久の本番確認

配信は単一の in-flight HTTP リクエストに依存させず、制御プレーンからの start/stop で駆動される常駐タスクとして走らせ、生存を制御プレーンが監視する構成にする。
